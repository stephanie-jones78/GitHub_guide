Before you touch
a single command.

Most people learn GitHub backwards. They start with commands before they understand what the commands are doing. Then they get confused, something breaks, and they quit. We're not doing that.

This section is the only one with no commands. Just the mental model. Get this right and everything else makes sense automatically.

What GitHub actually is

GitHub is two separate things that people constantly conflate:

⬡
FIRST THING
Git — the version control system
Git runs on your computer. It's software that tracks every change you make to your code over time. Every save is recorded. You can rewind to any point in history. Think of it as an infinitely detailed undo history — except it's permanent, searchable, and can handle hundreds of thousands of files.
↓ tap to expand
☁
SECOND THING
GitHub — the cloud storage and portfolio
GitHub is a website where you store copies of your Git history in the cloud. It's where other people can see your code, where you back it up, and where you deploy live websites from. Git is the tool. GitHub is the destination. They're related but distinct.
↓ tap to expand
// THE ANALOGY

Every game you play has a save file that records your exact state — your roster, your stats, your progress. Git is the save system. It runs locally on your machine and records every meaningful change. GitHub is the cloud backup — like syncing that save to the cloud so it can't be lost, so your teammates can load it, and so anyone who wants to see your career stats can look them up.

Why every serious developer uses it

Three reasons that matter for where you are right now:

✓
You can never permanently break anything
Every version of your code is saved. If you write something that destroys everything, you revert to yesterday's version in one command. No more "I accidentally deleted it all" stories.
✓
Your work becomes a public proof of skill
A GitHub profile with real projects on it is worth more than any certificate. When you cold-email a professor, apply to a program, or pitch an idea — your GitHub is what people check. Claims are cheap. Code is real.
✓
It's how 100 million developers actually work
Google, NASA, every open source project you've ever used — all of it lives on GitHub. Learning this now means you speak the same language as every professional developer on the planet.

One thing to internalize: GitHub isn't a feature you add when you're "ready." It's the first thing you set up. Even a project with one file deserves a repo. The habit of committing is more valuable than any individual project.

// SECTION 02

Six concepts.
Tap each one.

Every GitHub command you'll ever run maps back to one of these six ideas. Understand these and you'll be able to figure out unfamiliar commands on your own. Skip these and you'll copy-paste forever without knowing what anything does.

📁
CONCEPT 01
Repository (repo)
A repository is a project folder that Git is tracking. It contains all your code files plus a hidden folder called .git that stores the entire history of every change ever made. One project = one repo. You'll have many repos over time — one per project.
↓ tap to expand
📸
CONCEPT 02
Commit
A commit is a saved snapshot of your code at a specific point in time. Every commit has a unique ID (a hash like a3f9c2d), a message you write describing what changed, and a timestamp. Think of commits as the individual frames in a film — your project's complete history, frame by frame. Commit often. Each commit should represent one logical change.
↓ tap to expand
🎭
CONCEPT 03
Staging area
Before you commit, you choose exactly which changes to include. That selection process is called staging. You use git add to move changes into the staging area, then git commit to lock them in. This lets you make many edits but commit them in logical, organized groups rather than all at once.
↓ tap to expand
🌿
CONCEPT 04
Branch
A branch is a separate, parallel copy of your code where you can experiment freely without affecting your main working version. Your primary branch is called main. When you want to try a new feature, you create a branch, build it there, then merge it back when it's ready. Branches are free to create and instantly disposable.
↓ tap to expand
⬆
CONCEPT 05
Push and Pull
Push sends your local commits up to GitHub. Pull brings changes from GitHub down to your machine. These are the two directions of sync. When you work alone: push when you're done for the day. When working with others: pull before you start to get their latest changes, push when you're done to share yours.
↓ tap to expand
🔀
CONCEPT 06
Merge and Pull Request
Merging combines two branches back into one. A pull request (PR) is a formal proposal to merge — it shows exactly what changed, lets people review and comment, and creates a clear record of why changes were made. PRs are standard in professional development and become your portfolio of decision-making, not just code output.
↓ tap to expand

Knowledge check

You've been working on a new feature all afternoon. Your main branch is clean and working. Where should you be building?
Directly on main — it's my project, nobody else is using it
On a new branch — then merge it back when it's done and tested
A completely separate repo — less risk of breaking things
// SECTION 03

One-time setup.
Do this once. Never again.

Everything here runs once. After this your machine is permanently configured — every project you ever create benefits from this setup automatically.

// IN ORDER — DON'T SKIP STEPS
01
Create your GitHub account
›

Go to github.com. Your username becomes part of your public URL — github.com/yourname — so choose something professional. Your name, a clean handle. You can't easily change it later without breaking existing links.

Use your real email. Verify it. That's all you do on the website for now.

→ github.com/signup
02
Install Git on your machine
›

Git is the local program that actually tracks your code. Open Terminal and check if it's already there:

git --version

Mac: If not installed, macOS will prompt you to install via Xcode Command Line Tools — accept and wait about 2 minutes.

Windows: Download from git-scm.com and run the installer with all defaults.

03
Tell Git who you are
›

Every commit you make is permanently stamped with your name and email. This is how the record knows it was you. Use the same email as your GitHub account.

git config --global user.name "Your Name"
git config --global user.email "you@email.com"
git config --global init.defaultBranch main
04
Set up SSH — your secure handshake with GitHub
›

Why this exists: GitHub stopped accepting passwords from the terminal in 2021. Your account password will never work there — it's intentionally blocked. The terminal needs a different form of authentication: SSH keys.

What SSH actually is: SSH creates two mathematically linked keys — a private key that stays on your machine and a public key you give to GitHub. When you push code, your machine proves its identity by solving a challenge that only the holder of the private key can solve. GitHub verifies it using the public key you registered. No password. No prompts. Just works.

The two files you'll see:

After generating, your ~/.ssh/ folder will have two files with the same name:

id_ed25519 — your private key. Never share this. Never copy this anywhere. It stays on your machine only.

id_ed25519.pub — your public key. This is what you give to GitHub. Safe to share.

What the outputs mean: When you generate the key, Terminal prints random art and a fingerprint starting with SHA256:. That's just a visual confirmation the key was created — it's not the key itself. The actual key is in the .pub file.

05
Add your public key to GitHub
›

This is the step that connects your machine to your account. You do it once.

Copy your public key to clipboard:

cat ~/.ssh/id_ed25519.pub | pbcopy

Nothing visible happens — it's in your clipboard. Now on GitHub:

Profile photo → Settings → SSH and GPG keys → New SSH key → paste → Add SSH key

The key you paste will be one long line starting with ssh-ed25519 followed by a long string and ending with your email. If what you're pasting starts with SHA256 — that's the fingerprint, not the key. Run cat ~/.ssh/id_ed25519.pub and copy that output instead.

Verify it worked:

ssh -T git@github.com

You should see: Hi username! You've successfully authenticated. That's it. You're set.

Passwords vs SSH vs Tokens — the one thing to know: GitHub accounts use passwords to log into the website. But the terminal uses SSH keys. These are separate systems intentionally — the terminal has its own authentication so that even if someone learns your account password, they still can't push code from a machine that doesn't have your key.

If you're on Windows or pbcopy doesn't work, use this instead to print the key, then manually select and copy it:
cat ~/.ssh/id_ed25519.pub
Copy the entire line — from ssh-ed25519 to the end of your email. That's your public key.

What makes a good commit message

Your commit messages are a permanent log of your thinking. Write them like you're leaving notes for yourself six months from now — because you are.

✗
BAD COMMITS
Don't write these
"fix" "stuff" "changes" "asdfgh" "finally works" These tell you nothing. In three months you'll have no idea what "fix" refers to or why it was needed.
✓
GOOD COMMITS
Write these instead
"add user login form with validation" "fix score counter resetting on wrong answer" "refactor API calls into separate module" "add README with setup instructions" One idea per commit. Active verb. What changed and optionally why.

// SECTION 06

GitHub stores code.
Pages makes it live.

These are two different things that live in the same place — and confusing them is the #1 reason people get stuck at this stage.

📦
GITHUB — STORAGE
Shows raw code
When you push a file to GitHub, you can view the raw source code on github.com. An HTML file shows you the HTML. A Python file shows you the Python. GitHub does not run or render files — it stores and displays them as text.
🌐
GITHUB PAGES — HOSTING
Serves a live webpage
GitHub Pages is a hosting service built on top of your repo. Enable it and GitHub takes your index.html, serves it as a real webpage at a public URL, and runs all your CSS and JavaScript. The same file that showed raw code on GitHub becomes a fully interactive site anyone can open on any device.

The complete loop: build it locally → push to GitHub → Pages serves it live → share the URL. Every push to main updates the live site automatically within 60 seconds. No re-deploying, no extra steps.

What Pages can and cannot host

✓
WORKS ON PAGES
Static files
HTML, CSS, JavaScript — fully rendered and interactive. Single-page apps, tools, games, calculators, dashboards. This guide itself is a perfect example — one HTML file, full interactivity, hosted free on Pages.
✗
DOES NOT WORK ON PAGES
Server-side code
Python, Node.js, databases, server APIs. Anything that needs to execute code before responding to a user. When you hit this wall — and you will — use Vercel, Railway, or Render (all have free tiers). For now, Pages handles everything you're building.

The one rule that controls everything

GitHub Pages looks for exactly one file to serve as your homepage: index.html — lowercase, at the root of your repo. Not app.html. Not my-project.html. Not index.HTML. It must be index.html. If that file doesn't exist in your repo, Pages has nothing to serve and your URL will show the README instead.

Before you enable Pages — verify this first: go to github.com/yourusername/your-repo and confirm you can see index.html listed as a file. If you can't see it there, it hasn't pushed yet. Fix that before touching the Pages settings.

Complete deploy sequence — in order, nothing skipped

01
Name your HTML file index.html
Your main HTML file must be named exactly index.html. If it has any other name, rename it now before doing anything else. The rename and the push must happen together — renaming locally without pushing means GitHub still has the old filename.
02
Rename, commit, and push as one sequence
Do not rename and stop. Do not rename, then push later. Run all four commands back to back:
03
Verify the file is on GitHub before enabling Pages
Open github.com/yourusername/your-repo in your browser. Look at the file list. You must see index.html there. If you see the old filename — the push didn't complete. Re-run git push and check again. Do not move to the next step until index.html is visible on GitHub.
04
Enable GitHub Pages
On your repo page → Settings tab → Pages in the left sidebar → Source: Deploy from a branch → Branch: main → Folder: / (root) → click Save.

If Save is greyed out: change the Branch dropdown to None, click Save, then change it back to main, set folder to / (root), click Save again. This resets the state and unsticks the button.
05
Wait 60 seconds — then open your URL
Refresh the Pages settings page. You'll see a green banner with your live URL:

https://yourusername.github.io/repo-name/

Open it. Your site should be fully rendered and interactive. If it still shows the README — wait another 30 seconds and hard refresh (Cmd+Shift+R on Mac). GitHub sometimes takes a moment to propagate the first deploy.
06
Update your README with the live URL
Open your README, replace any placeholder link with your real Pages URL. Then push that change. Anyone who visits your repo on GitHub now sees the live link immediately at the top of the README.

// SECTION 07

When things break.
Every error, explained.

These are the exact errors you will hit. Not hypothetical ones — the real ones that stop people cold. Each one has one cause and one fix.

// ERRORS IN THE ORDER YOU'LL ENCOUNTER THEM
!
Authentication failed / password rejected in terminal
›

What happened: GitHub removed password authentication from the terminal in 2021. Your account password works on the website only. The terminal is a different authentication system entirely.

Fix: Set up SSH keys (Section 3) or use a Personal Access Token. For PAT: GitHub → Settings → Developer settings → Personal access tokens → Tokens (classic) → Generate new token → check repo → copy the token → paste it where the terminal asks for a password.

!
SSH key invalid — must supply OpenSSH public key format
›

What happened: You copied the key fingerprint (SHA256:...) instead of the actual public key. The fingerprint is printed when the key is generated as a confirmation — it is not the key itself.

Fix: Run cat ~/.ssh/id_ed25519.pub and copy the output. It must start with ssh-ed25519 not SHA256. On Mac, use cat ~/.ssh/id_ed25519.pub | pbcopy to copy cleanly to clipboard.

!
fatal: Updating an unborn branch with changes added to the index
›

What happened: You ran git push before making your first commit. Git has nothing to push yet — the branch doesn't exist until the first commit creates it.

Fix:

git add .
git commit -m "initial commit"
git push -u origin main
!
error: failed to push some refs
›

What happened: GitHub has commits your local machine doesn't have — usually because GitHub auto-created a README or .gitignore when you created the repo. Your histories have diverged.

Fix:

git pull origin main --rebase
git push

If that returns "refusing to merge unrelated histories":

git pull origin main --allow-unrelated-histories
git push

Prevention: When creating a repo on GitHub, leave all checkboxes unchecked — no README, no .gitignore, no license. Start completely empty.

!
error: remote origin already exists
›

What happened: You ran git remote add origin twice. The remote is already set — you can't add it again.

Fix: Skip the git remote add line and run:

git branch -M main
git push -u origin main

To check what remote is currently set: git remote -v

!
GitHub Pages shows README instead of my site
›

What happened: One of three things — your file isn't named index.html, the rename didn't push to GitHub, or Pages hasn't finished deploying yet.

Fix — check in this order:

1. Go to github.com/yourusername/your-repo and look at the file list. Do you see index.html? If not — it didn't push. Run git push again.

2. If index.html is there but the site still shows README — wait 60 more seconds and hard refresh: Cmd+Shift+R on Mac.

3. If it still shows README — go to Settings → Pages and confirm it's set to deploy from main branch, / (root) folder.

!
GitHub Pages Save button is greyed out
›

What happened: GitHub's Pages UI sometimes gets into a state where Save stays disabled even when the correct options are selected. It's a UI quirk, not an error in your configuration.

Fix: Change Branch dropdown to None → click Save → change back to main → set folder to / (root) → click Save. Resetting to None and back unsticks it.

!
404 error on GitHub Pages URL
›

What happened: Either Pages isn't enabled yet, the file isn't named index.html, or the URL has a typo.

Fix — verify all three:

1. Settings → Pages — confirm it shows a green "Your site is live at..." banner

2. Confirm index.html exists in your repo on github.com

3. Your URL format is: https://USERNAME.github.io/REPO-NAME/ — username and repo name are case sensitive, must match exactly

!
Interactive elements don't work when opening file locally on iPhone
›

What happened: iOS Safari blocks JavaScript on local files opened from storage (file:// protocol) as a security measure. The layout renders but interactivity is disabled.

Fix: This is exactly what GitHub Pages solves. Host the file there and share the github.io URL instead of the file itself. The URL runs on https:// which iOS Safari trusts fully.

Build in public.
Your reputation compounds.

GitHub isn't just storage. It's a live record of your development as a builder. The people who use it most effectively treat their commit history like a public log — not just of what they built, but of how they think.

Habits that separate serious builders from hobbyists

✓
Commit every working session
Even if you only wrote 10 lines. Even if it's not finished. A commit at the end of every session means your history reflects actual work, not just completion moments. Consistency compounds over time.
✓
Write your README before you consider the project done
A project without a README is a project only you can use. The README is the first thing any potential collaborator, employer, or professor reads. Write it like your repo is receiving visitors — because it is.
✓
Don't wait for finished — push works in progress
Unfinished work on GitHub is still evidence of work. A half-built tool with 40 commits and a clear README is more impressive than a finished project that appeared in one commit. The process is part of the portfolio.
✓
Pin your best repos to your profile
GitHub lets you pin up to 6 repositories to the top of your profile. Choose the ones that best represent what you can do. Update them as your work improves. Your profile is a living portfolio — treat it like one.
✓
Use Issues to track your own ideas
GitHub Issues aren't just for bugs in team projects. On your own repos, use them to track features you want to build, problems you hit, and ideas you have. It documents your thinking — and it looks professional when someone looks at your repo.
✓
Read other people's code
Every open source project on GitHub is readable. Find a tool you use, go to its repo, read the code. You don't have to understand it all. The habit of reading production code — how real engineers structure, name, and comment things — improves your own work faster than any tutorial.

The cold email that works: "I built X. Here's the link. I'm interested in Y. Here's what I've been working on: [github.com/yourprofile]." That combination — a deployed project and an active profile — opens doors that a resume alone never does.
