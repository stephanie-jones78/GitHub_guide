Before you touch
a single command.

Most people learn GitHub backwards. They start with commands before they understand what the commands are doing. Then they get confused, something breaks, and they quit. We're not doing that.

This section is the only one with no commands. Just the mental model. Get this right and everything else makes sense automatically.

What GitHub actually is

GitHub is two separate things that people constantly conflate:

⬡
FIRST THING
Git — the version control system
Git runs on your computer. It's software that tracks every change you make to your code over time. Every save is recorded. You can rewind to any point in history. Think of it as an infinitely detailed undo history — except it's permanent, searchable, and can handle hundreds of thousands of files.
↓ tap to expand
☁
SECOND THING
GitHub — the cloud storage and portfolio
GitHub is a website where you store copies of your Git history in the cloud. It's where other people can see your code, where you back it up, and where you deploy live websites from. Git is the tool. GitHub is the destination. They're related but distinct.
↓ tap to expand
// THE ANALOGY

Every game you play has a save file that records your exact state — your roster, your stats, your progress. Git is the save system. It runs locally on your machine and records every meaningful change. GitHub is the cloud backup — like syncing that save to the cloud so it can't be lost, so your teammates can load it, and so anyone who wants to see your career stats can look them up.

Why every serious developer uses it

Three reasons that matter for where you are right now:

✓
You can never permanently break anything
Every version of your code is saved. If you write something that destroys everything, you revert to yesterday's version in one command. No more "I accidentally deleted it all" stories.
✓
Your work becomes a public proof of skill
A GitHub profile with real projects on it is worth more than any certificate. When you cold-email a professor, apply to a program, or pitch an idea — your GitHub is what people check. Claims are cheap. Code is real.
✓
It's how 100 million developers actually work
Google, NASA, every open source project you've ever used — all of it lives on GitHub. Learning this now means you speak the same language as every professional developer on the planet.

One thing to internalize: GitHub isn't a feature you add when you're "ready." It's the first thing you set up. Even a project with one file deserves a repo. The habit of committing is more valuable than any individual project.

// SECTION 02

Six concepts.
Tap each one.

Every GitHub command you'll ever run maps back to one of these six ideas. Understand these and you'll be able to figure out unfamiliar commands on your own. Skip these and you'll copy-paste forever without knowing what anything does.

📁
CONCEPT 01
Repository (repo)
A repository is a project folder that Git is tracking. It contains all your code files plus a hidden folder called .git that stores the entire history of every change ever made. One project = one repo. You'll have many repos over time — one per project.
↓ tap to expand
📸
CONCEPT 02
Commit
A commit is a saved snapshot of your code at a specific point in time. Every commit has a unique ID (a hash like a3f9c2d), a message you write describing what changed, and a timestamp. Think of commits as the individual frames in a film — your project's complete history, frame by frame. Commit often. Each commit should represent one logical change.
↓ tap to expand
🎭
CONCEPT 03
Staging area
Before you commit, you choose exactly which changes to include. That selection process is called staging. You use git add to move changes into the staging area, then git commit to lock them in. This lets you make many edits but commit them in logical, organized groups rather than all at once.
↓ tap to expand
🌿
CONCEPT 04
Branch
A branch is a separate, parallel copy of your code where you can experiment freely without affecting your main working version. Your primary branch is called main. When you want to try a new feature, you create a branch, build it there, then merge it back when it's ready. Branches are free to create and instantly disposable.
↓ tap to expand
⬆
CONCEPT 05
Push and Pull
Push sends your local commits up to GitHub. Pull brings changes from GitHub down to your machine. These are the two directions of sync. When you work alone: push when you're done for the day. When working with others: pull before you start to get their latest changes, push when you're done to share yours.
↓ tap to expand
🔀
CONCEPT 06
Merge and Pull Request
Merging combines two branches back into one. A pull request (PR) is a formal proposal to merge — it shows exactly what changed, lets people review and comment, and creates a clear record of why changes were made. PRs are standard in professional development and become your portfolio of decision-making, not just code output.
↓ tap to expand

Knowledge check

You've been working on a new feature all afternoon. Your main branch is clean and working. Where should you be building?
Directly on main — it's my project, nobody else is using it
On a new branch — then merge it back when it's done and tested

One-time setup.
Do this tonight.

Everything in this section happens once. After this, your machine is configured and you never have to do it again.

// SETUP CHECKLIST — IN ORDER
01
Create a GitHub account
›

Go to github.com. Choose a username you'd be comfortable showing an employer — your name, a clean handle, something professional. This is your public identity as a developer. You can't easily change it later without breaking links.

Use your real email. Verify it. That's it for the website for now.

→ github.com/signup
02
Install Git on your machine
›

Git is a program that runs locally. On Mac, open Terminal and run:

git --version

If it's not installed, macOS will prompt you to install it automatically via Xcode Command Line Tools. Accept and wait. On Windows, download Git from git-scm.com. Run the installer with all defaults.

03
Configure your identity
›

Every commit you make is signed with your name and email. Run these two commands — use the same email as your GitHub account:

04
Set up SSH authentication
›

SSH lets you push to GitHub without typing your password every time. Generate a key pair — a private key that stays on your machine and a public key you give to GitHub.

After generating, copy your public key (cat ~/.ssh/id_ed25519.pub) and paste it into GitHub → Settings → SSH and GPG keys → New SSH key.

One-time setup. Never deal with passwords again.

05
Set default branch name to main
›

Git used to default to calling the primary branch master. GitHub now uses main. Make them match to avoid confusion:

3git config --global user.email "you@example.com"

15git config --global --list

You only do this once. From now on your machine knows who you are and can talk to GitHub securely. Every project you ever create will use this configuration automatically.

What makes a good commit message

Your commit messages are a permanent log of your thinking. Write them like you're leaving notes for yourself six months from now — because you are.

✗
BAD COMMITS
Don't write these
"fix" "stuff" "changes" "asdfgh" "finally works" These tell you nothing. In three months you'll have no idea what "fix" refers to or why it was needed.
✓
GOOD COMMITS
Write these instead
"add user login form with validation" "fix score counter resetting on wrong answer" "refactor API calls into separate module" "add README with setup instructions" One idea per commit. Active verb. What changed and optionally why.

Branches.
Your most underused tool.

Most beginners do everything on main and wonder why their code keeps breaking. Branches solve this. They're free to create, instant to switch between, and disposable when you're done.

// THE ANALOGY

Your starting lineup is your main branch — it's what's actually running, what performs in public, what everything depends on. When a coach wants to try a new rotation or test a player in a different position, they don't rebuild the starting lineup around an experiment. They run it in practice first. Branches are practice. Build it there, prove it works, then run it in the real game.

// SECTION 06

Publish your work.
Free. Tonight.

GitHub Pages lets you deploy a live website directly from your repository. Free, no server, no hosting account. If your project has an index.html, it can be live at yourusername.github.io/project-name in under five minutes.

// THE ANALOGY

There's a difference between a player who practices in the gym and a player who performs in a game. Both are working. Only one has a record. GitHub Pages is your game tape. It's the difference between "I built a quiz game" and "here's the link, try it right now." The link ends every conversation.

01
Make sure your project has an index.html
GitHub Pages serves whatever is at the root of your repo as a website. Your main HTML file needs to be named index.html. If you have a JavaScript app or a different structure, there are ways to configure it — but start with a plain HTML project to get the pattern.
02
Push your project to GitHub
If you followed Section 4, your repo is already on GitHub. Make sure your latest code is pushed: git add . && git commit -m "prepare for deployment" && git push
03
Enable GitHub Pages in Settings
On your repo page on github.com → Settings tab → Pages in the left sidebar → Source: Deploy from a branch → Branch: main → folder: / (root) → click Save.
04
Wait 60 seconds. Your site is live.
GitHub will show you the URL: https://yourusername.github.io/your-repo-name. It takes about a minute to deploy on the first push, then updates automatically every time you push new commits. Share the link. That's real.
05
Add a custom domain later (optional)
If you eventually want yourname.com instead of github.io, buy a domain ($10–15/yr on Namecheap or Cloudflare), then configure it in the same Pages settings. GitHub handles the HTTPS certificate automatically. No cost, no server.

Build in public.
Your reputation compounds.

GitHub isn't just storage. It's a live record of your development as a builder. The people who use it most effectively treat their commit history like a public log — not just of what they built, but of how they think.

Habits that separate serious builders from hobbyists

✓
Commit every working session
Even if you only wrote 10 lines. Even if it's not finished. A commit at the end of every session means your history reflects actual work, not just completion moments. Consistency compounds over time.
✓
Write your README before you consider the project done
A project without a README is a project only you can use. The README is the first thing any potential collaborator, employer, or professor reads. Write it like your repo is receiving visitors — because it is.
✓
Don't wait for finished — push works in progress
Unfinished work on GitHub is still evidence of work. A half-built tool with 40 commits and a clear README is more impressive than a finished project that appeared in one commit. The process is part of the portfolio.
✓
Pin your best repos to your profile
GitHub lets you pin up to 6 repositories to the top of your profile. Choose the ones that best represent what you can do. Update them as your work improves. Your profile is a living portfolio — treat it like one.
✓
Use Issues to track your own ideas
GitHub Issues aren't just for bugs in team projects. On your own repos, use them to track features you want to build, problems you hit, and ideas you have. It documents your thinking — and it looks professional when someone looks at your repo.
✓
Read other people's code
Every open source project on GitHub is readable. Find a tool you use, go to its repo, read the code. You don't have to understand it all. The habit of reading production code — how real engineers structure, name, and comment things — improves your own work faster than any tutorial.
// THE ANALOGY

The best players in the league have a body of work that speaks for itself — every game in the record, every stat tracked, every highlight accessible. Your GitHub is that record. The green squares on your contribution graph are your game log. They don't lie about whether you're putting in the work. Build the kind of profile that speaks before you have to.

The cold email that works: "I built X. Here's the link. I'm interested in Y. Here's what I've been working on: [github.com/yourprofile]." That combination — a deployed project and an active profile — opens doors that a resume alone never does.
